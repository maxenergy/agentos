# AgentOS 产品需求文档（PRD）

## 1. 产品概述

### 1.1 产品名称
AgentOS

### 1.2 产品定位
AgentOS 是一个可持续成长的 AI Agent 运行时内核。  
它不是单一模型封装器，也不是简单工具调用器，而是一个具备：

- 自身基础能力
- 外部工具接入能力
- 外部专家协作能力
- 执行复盘与成长能力

的系统内核。

### 1.3 产品目标
构建一个“像有经验的工程师一样工作”的系统：

- 自己能做基础工作
- 会借助外部工具
- 会在必要时请外部专家协助
- 能从历史执行中总结经验，逐步提升效率和质量

---

## 2. 目标用户

### 2.1 主要用户
- AI Infra 工程师
- Agent 框架研发者
- 自动化平台开发者
- 高级个人开发者
- 需要搭建长期 Agent 系统的小团队

### 2.2 典型用户诉求
- 我想做一个稳定的 Agent 内核，而不是脚本堆砌
- 我想统一接入各种 CLI / AI CLI / 插件
- 我想用多个 coding agent 协作，而不是只绑死一家
- 我想让系统逐步积累经验和工作流
- 我想让系统具备工程级安全、路由和审计能力

---

## 3. 核心价值主张

### 3.1 对用户的价值
AgentOS 提供一种比“单个 LLM + 一堆工具”更高级的系统形态：

- 更稳定：核心运行时尽量少依赖、强约束
- 更灵活：外部能力统一接入
- 更可控：有权限、策略、审计、评分
- 更会成长：记忆、Workflow、经验总结是一等公民

### 3.2 与普通 Agent 系统的区别

| 维度 | 普通 Agent | AgentOS |
|------|------------|---------|
| 内核 | 往往较重或较散 | 最小稳定内核 |
| 工具接入 | 零散 | 统一 Skill/CLI/Plugin/Agent |
| 多代理协作 | 弱 | 明确支持二级代理 |
| 记忆 | 以聊天历史为主 | 结构化任务记忆 |
| 演化 | 弱 | Workflow 与评分驱动 |
| 安全控制 | 往往薄弱 | 策略与边界明确 |

---

## 4. 核心功能

### 4.1 核心运行时
系统必须具备：

- Task 接收与执行循环
- 上下文加载与任务调度
- Skill 与 Agent 路由
- 结果汇总与失败恢复
- 生命周期与会话管理
- 审计日志

### 4.2 Skill 系统
系统必须支持以下 Skill 类型：

1. Atomic Skill
   - 最小基础能力
   - 例如 file_read / file_write / http_fetch

2. Workflow Skill
   - 多步骤组合技能
   - 由历史经验或人工定义生成

3. CLI Skill
   - 将现有 CLI 工具纳入统一能力体系

4. Agent Skill
   - 将外部 AI Agent 以统一接口接入系统

### 4.3 外部工具能力
系统必须支持：

- 通用 CLI 工具接入
- 多语言插件接入
- 现有 AI CLI 接入
- session 型工具接入
- 结构化输出与解析

### 4.4 二级代理系统
系统必须支持外部“专家型代理”：

- Claude Code
- Codex CLI
- Gemini CLI
- Qwen Code
- 预留一个实验位给未来新代理或自研代理

并要求：

- 数量受控（建议不超过 5 个主力位）
- 有统一 Agent Adapter 接口
- 有路由与评分机制
- 可支持单次调用和会话模式

### 4.5 记忆与成长系统
系统必须支持：

- 任务记录
- 步骤记录
- 成功与失败案例存档
- Workflow 自动抽取
- Agent/Skill 评分
- 路由策略优化

### 4.6 认证与凭据系统
系统必须支持：

- OAuth
- API key
- CLI 登录态复用
- 多账号、多 profile
- 安全存储与刷新

### 4.7 安全与策略控制
系统必须支持：

- 工作区隔离
- 权限声明
- 高危命令限制
- 外部进程隔离
- 日志审计
- 风险等级控制

---

## 5. MVP 范围

第一版仅需实现：

### 5.1 内核
- Agent loop
- Skill Registry
- Agent Registry
- Router
- Memory 基础版
- Policy 基础版

### 5.2 内建基础 Skill
至少包含：

- file_read
- file_write
- file_patch
- http_fetch
- workflow_run

### 5.3 外部能力
- CLI Host
- 1 个二级代理接入
- 认证子系统基础版

### 5.4 记忆与演化
- 任务事件日志
- Workflow 自动生成的基础骨架
- Skill/Agent 基础评分

---

## 6. 非功能需求

### 6.1 稳定性
- 内核应尽量无状态或低状态
- 高风险能力不直接运行在主进程
- 每个外部能力有清晰超时与隔离

### 6.2 安全性
- 所有高危操作必须经过策略层
- 凭据不得明文写入日志
- 工作区操作范围必须可限制

### 6.3 可扩展性
- 新 Skill 接入不应大改核心
- 新 provider/auth mode 通过 adapter 扩展
- 新二级代理通过统一接口接入

### 6.4 可观测性
- 执行日志清晰
- 路由过程可解释
- Skill/Agent 状态可查看
- 失败原因可追踪

---

## 7. 成功指标

### 7.1 工程指标
- 新增一个 CLI Skill 的接入成本较低
- 新增一个二级代理的接入成本较低
- 核心模块具有清晰接口与测试边界

### 7.2 产品指标
- 系统能稳定完成基础开发任务
- 能从任务中形成可复用 Workflow
- 随着使用次数增加，路由质量提升

### 7.3 演化指标
- 高频任务能够被自动抽取为 Workflow
- 高成功率 Skill 和 Agent 被优先路由
- 低表现能力被降级或灰度下线
